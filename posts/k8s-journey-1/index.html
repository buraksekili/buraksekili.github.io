Pods

Even though the pods might contain multiple containers, all are running on a single worker node; so, it doesn’t work on multiple worker nodes.

Pods are basic units of scaling in k8s, so instead of scaling containers individually, k8s scales pods.

Resources

  1. Even you have multiple containers within a pod, each container shares the same IP address, port space, and Linux namespaces. So, be careful about port conflicts within a pod. But different pods do not have such a problem, since containers in a pod run in the same Network, not pods in a worker node.

  2. For local file system I/O, you probably do not have a common filesystem because most containers run upon different file systems. You can use volumes to share a common file system.

  3. Pods in a k8s cluster share network address space, so each pod is able to access other pods through IP address.

    • Since pods in a cluster share network address space, there is no need for network address translation (NAT).

Pods with multiple containers? Organization of containers

You probably might want to split your application into multiple containers (e.g., domain-specific containers, email services, notification services). So, you might ask the following questions to yourself to decide on whether you need to put multiple containers in a pod.

  1. Can you scale them together?
  2. Do they construct a single entity?
  3. Can you run together (tightly coupled manner)?

Creating pods? YAML

To simply put, you just create a YAML or JSON manifests (YAML is a common way of defining pods) and send them REST API of the k8s (master node). Also, you can use kubectl for creating a pod but in general, you will need to define a YAML file to describe various pod attributes.

YAML from Kubernetes GitHub 1 and Kubernetes in Action 2

There are three major parts of this YAML file:

  1. Metadata: As name implies, basic metadata regarding the pod.
  2. Spec: Actual description of the pod including containers & volumes.
  3. Status: Current health of the pod.
